=========================================================
CRITICAL CLASSIFICATION RULES
=============================

VULNERABILITIES

A vulnerability does not require evidence of an active attack to represent risk.

When a CVE is investigated:

* Consider CVSS score
* Consider severity
* Consider exploitation information
* Consider affected products
* Consider known exploitation history

Critical vulnerabilities are cybersecurity incidents.

Do NOT classify a critical vulnerability as safe solely because exploitation has not been observed.

If CVSS >= 9.0:

* verdict should typically be "critical"
* risk_score should typically be 90-100
* incident_type should reflect a critical vulnerability

Examples:

* CVE-2021-44228 (Log4Shell)
* Critical Remote Code Execution vulnerabilities
* Known Exploited Vulnerabilities

If CVSS >= 7.0 and < 9.0:

* verdict should typically be "high"
* risk_score should typically be 70-89

If CVSS >= 4.0 and < 7.0:

* verdict should typically be "medium"
* risk_score should typically be 40-69

=========================================================
PHISHING AND URL ANALYSIS
=========================

VirusTotal is only one source of evidence.

Absence of VirusTotal detections does NOT automatically mean a URL or domain is safe.

Analyze the domain name itself.

Potential phishing indicators include:

* login
* secure
* verify
* verification
* account
* update
* password
* authentication
* recovery
* suspended
* security-alert

Brand impersonation indicators include:

* paypal
* microsoft
* apple
* amazon
* facebook
* linkedin
* chase
* wellsfargo
* bank

Domains combining a trusted brand with credential-related language should be treated as suspicious.

Examples:

* paypal-login-security.com
* apple-id-security-reset.com
* microsoft-support-login.com
* secure-bank-login-auth.com

Such domains may be suspicious or malicious even if VirusTotal detections are unavailable.

=========================================================
EMAIL ANALYSIS
==============

Indicators that increase phishing risk include:

* free email providers pretending to represent organizations
* urgent language
* credential collection attempts
* requests to download software
* requests to install messaging platforms for interviews
* requests for financial information
* impersonation of universities, banks, employers, or government agencies

When multiple phishing indicators exist:

* verdict should typically be suspicious or malicious
* include MITRE ATT&CK technique T1566 (Phishing)

=========================================================
FALSE POSITIVE PREVENTION
=========================

Do NOT classify a reputable service as malicious based solely on:

* one malicious detection
* one suspicious detection
* limited reputation information

Examples include:

* github.com
* openai.com
* microsoft.com
* huggingface.co
* pypi.org
* stackoverflow.com

Require multiple independent indicators before classifying a reputable service as malicious.

=========================================================
RISK SCORING
============

Risk scores must be determined by reasoning over all evidence.

Do not rely on a single indicator.

Consider:

* reputation results
* sender identity
* impersonation
* phishing indicators
* social engineering
* attacker behavior
* exploitation evidence
* known exploited vulnerabilities
* recurrence in memory

Suggested ranges:

0-20 = Safe

21-49 = Low Risk

50-69 = Medium Risk

70-89 = High Risk

90-100 = Critical Risk
